개인 프로젝트만 진행하다가 처음 하는 팀 프로젝트여서 그런지 협업 관련해서 초반에 github 관련해서 실수했던 부분도 있었고 혼란스러웠지만 새로 배우는게 많아 즐겁게 개발했던 한 주였다.
같은 팀 팀원분들도 다들 열정적으로 진행해주셔서 나도 더 열심히 했던 한주였다.

### 기능 구현

#### 1. 페이지 UI 구현

현재 맡은 페이지의 UI들을 대부분 구현했다.
구현할 때 살짝 당황했던 부분은 Figma에 css가 정의되어있지 않고 이미지로 UI를 작성한 부분이 일부 있었기 때문이다.
이 부분에 대해 멘토님한테 여쭤보니 전체 디자인에서 크게 벗어나지 않는 선에서 본인 재량으로 css를 작성하면 된다고 하셨다.
그래서 내 재량으로 css가 정의되어있는 타 UI들과 잘 어울리도록 css를 작성했다.

#### 2. 검색 기능 구현

UI 구현을 마친 기능들을 하나씩 추가해나가기로 했다.
우선 검색 기능은 Form을 사용한 기능이므로 `react-hook-form`을 이용해 form data를 관리하도록 만들었다.
이를 통해 form 최적화 및 `validation`이나 `error`처리 등을 쉽게 해줄 수 있었다.
그 다음 이제 검색어를 입력해 리스트에서 해당 검색어에 대한 내용을 가져오는 기능을 추가했다.
하지만 문제가 하나 있었다면 우리는 리스트 데이터를 가져올 수 있는 api등을 받지 못했다는 것.
그래서 `mock data`를 이용해 임시로 내가 데이터를 만들어서 임시 데이터 리스트에서 검색할 수 있도록 구현했다.

#### 3. checkbox 기능 구현 (단일 선택, 전체 선택)

검색 기능을 끝마친 다음 한 일은 checkbox기능을 구현한 일이었다.
이 부분은 지난주 사전직무교육에서 todo list를 만들 때와 동일해 쉽게 로직을 구현할 수 있었다.

여기에 추가된 부분은 바로 '전체 선택' 부분이었는데
이전에 회원가입 페이지를 만들면서 해당 기능을 만든 적이 있었지만 그 때에는 잘 몰라서 항목마다 이름을 부여해 여러 조건문을 추가해 만들었었다. 😥
그러다보니 코드가 굉장히 복잡했던 기억이 난다.

전체 선택은 check 관련 state를 prev => !prev함과 동시에 요소들 중에 선택되거나 선택되지 않은 요소가 있는지 확인하는 로직이 필요하다.
전체 선택을 체크했을 때 <b>하나의 요소가 true이면 그 true인 요소는 그대로 나머지만 false에서 true로.
그 반대라면 true였던 요소 하나만 false로 나머지는 그대로 false</b>로 두어야 한다.
예전의 나는 이걸 곧이곧대로 조건들을 추가해 순서대로 로직을 작성했었나보다.

<b>하지만 이번에 나는 다르다!</b> 해당 기능은 javascript의 `some`, `every` 메서드를 이용해 정말 간단하게
전체선택 기능을 구현할 수 있었다. 그 사이 javascript를 조금 더 공부하고 코딩테스트 문제를 조금 더 풀어봤던
게 도움이 되었나보다. 이렇게 지저분했던 로직이 하나씩 깔끔해지는 것을 보니 뿌듯했다.

```
useEffect(() => {
  const findUnCheckedItem = localRegister.result.every(
    (item) => item.isChecked
  );

  setIsCheckedAll(findUnCheckedItem);
}, [localRegister]);
```

```
const toggleCheckboxAll = () => {
  setIsCheckedAll((prev) => !prev);

  setLocalRegister((prev) => ({
    ...prev,
    result: prev.result.map((item) => ({
      ...item,
      isChecked: !isCheckedAll,
    })),
  }));
};
```

```
<Checkbox
  type='checkbox'
  onClick={toggleCheckboxAll}
  checked={isCheckedAll}
></Checkbox>
```

### 문제가 생겼던 부분과 해결 방법

#### 1. github 관련 이슈

프로젝트를 진행하면서 나를, 아니 우리 팀을 가장 괴롭혔던 부분은 다름 아닌 github였다.
혼자 프로젝트를 진행할 때는 코드가 충돌이 날 일이 없어서 몰랐는데 팀 프로젝트로 같은 repo에서 진행하다보니 충돌 이슈가 종종 발생했다.
우리는 충돌을 방지하기 위해 git commit flow를 정했지만 그럼에도 불구하고 피해갈 수 없었던 충돌 이슈.
온갖 블로그 서치와 지인찬스를 이용해 겨우 해결할 수 있었다.
항상 로컬에서 먼저 충돌 이슈를 해결하고 커밋하기 명심 또 명심하기!!

그리고 github 관련해서 이슈가 하나 더 있었는데 바로 대/소문자 이슈.
폴더를 소문자로 만들었어야 했는데 대문자로 만들어버린 것.. 😫😫
더욱 문제는 github는 대/소문자를 구분하지 않아서 내가 로컬에서 폴더명을 소문자로 바꾸고 commit하려고 해도 인식을 하지 못했다.
인터넷 서치를 통해 수소문해 발견한 방법은 git command를 이용해 폴더 이름을 임시 이름으로 바꾸고
다시 임시 이름에서 최종으로 사용할 이름으로 바꿔서 올리는 방법.
이건 또 vscode terminal에서 적용되지 않아서 git bash를 이용했다.

```
git mv Folder_name temp
git mv temp folder_name
```

이렇게 해서 간신히 이슈 해결 완료.
프로젝트 초반이라 협업하는 과정에서 이런 이슈들이 있었는데 다행이도 프로젝트를 진행하면서
발전해 github 충돌 관련해 이슈는 더이상 나오지 않게 되었다🥰❤

#### 2. API와 Mock data

내가 맡은 파트는 API 통신이 필요한 파트였지만 프로젝트 처음부터 API가 제공되지는 않았다.
하지만 마냥 백엔드에서 API가 오기까지 기다릴 수 없었다. 나도 개발할 기능이 많았고 또한
FE개발자가 개발할 때 BE개발자도 당연히 개발을 할 것이 아닌가. 심지어 우리는 FE팀원들밖에 없어서 BE부분은 멘토님께서 해주시는데
멘토님은 당연히 회사 일도 겸업하고 계셨다.
따라서 이 때 미리 Mock Data를 만들어 기능을 개발해 개발 속도도 단축되면서 효율적으로 개발할 수 있었다.
